Key Design Question: Extend vs. New Type

The platform already has short_answer as a question type. We have two approaches:

A

Extend short_answer with validation config

Add max_length, validation_type (none/regex/email/...), validation_regex, and normalize fields to the existing short_answer type's options. Also add a description (helper text) field to ALL question types. Postal code prefix is just one configuration of the generic short_answer type.

B

New postal_code_prefix question type

Create a dedicated postal_code_prefix question type with hardcoded behavior (max 3 chars, uppercase normalization, ^[A-Z][0-9][A-Z]$ regex). No generic validation config exposed to the admin UI — postal code is a first-class type alongside the existing ones.

My Recommendation: A

The spec explicitly says "add a short-answer question type" and asks for it to be "configurable per survey question." Making a generic short_answer with validation config is more useful long-term and avoids the code sprawl of a one-off type. The postal code prefix is just a preset configuration.
